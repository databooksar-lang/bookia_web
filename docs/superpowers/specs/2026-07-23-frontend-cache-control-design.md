# Frontend Cache-Control Design

## Objective

Ensure a deployed Bookia frontend updates automatically for returning users while retaining efficient caching for versioned static assets.

## Evidence

- The production image serves the SPA through Caddy, generated by `docker-entrypoint.sh`.
- Neither the static Caddyfile nor the generated Caddyfile declares `Cache-Control` policies.
- There is no service worker or PWA cache layer in the frontend.
- Vite emits JavaScript and CSS assets with content hashes in their filenames, so those files can safely be cached long-term.
- `index.html` references the latest hashed assets and `runtime-config.js`; serving a stale copy of either can keep a user on an older deployment or API configuration.

## Chosen Approach

Configure route-specific Caddy response headers in `docker-entrypoint.sh`:

- `/` and all SPA fallback responses (`index.html`) will use `Cache-Control: no-cache`. Browsers and intermediaries may store the response but must revalidate it before reuse.
- `/runtime-config.js` will use `Cache-Control: no-cache`, so API routing changes made at container startup are revalidated.
- `/assets/*` will use `Cache-Control: public, max-age=31536000, immutable`. Vite content hashes make this safe and avoid downloading unchanged JavaScript and CSS.
- Other static files will retain Caddy's existing behavior. This avoids imposing a one-year policy on unversioned public images, whose URLs may be reused.

## Delivery and Verification

Add a regression assertion to the existing Node test runner that reads `docker-entrypoint.sh` and verifies all three policies. Run the test suite and a production Vite build. After deployment, confirm from browser DevTools or `curl -I` that the production headers match this design.

## Non-goals

- Do not add a service worker or offline mode.
- Do not change the application API caching behavior.
- Do not change CDN or Railway settings outside this repository.
